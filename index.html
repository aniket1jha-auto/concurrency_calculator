The math

Every output comes from these.

The three solve modes

One equation — work = concurrency × duration — solved for a different unknown depending on what's fixed. Like distance = speed × time: give any two, get the third.

  • Hit a deadline — you know volume + SLA. Unknown = concurrency, and the scaling needed to reach it. The "what must we grow" question.
  • Set concurrency — concurrency is contracted or chosen. Unknown = duration. The "what happens at this number" question.
  • Cap the backend — backend is fixed and not being raised. Unknown = what's achievable. Concurrency auto-sets to min(backend, Azure); SIP and DIDs are sized to the cap so you never over-buy. The "what's possible inside a hard limit" question.
Modes 1 and 2 are inverses. Mode 3 is mode 2 with concurrency locked to your real ceiling instead of typed in. The scaling-plan table — current → required → delta — is the output that matters in all three.

Work & concurrency

attempts A = base × retry channel_hold = ASR × talk + (1−ASR) × ring (sec/attempt) total_work W = A × channel_hold (channel-seconds) useful sec/channel/day = window_hrs × 3600 × occupancy deadline D → concurrency C = ⌈ W / (D × useful_sec_per_day) ⌉ concurrency → duration_days = W / (C × occupancy × 3600) / window_hrs capped → C = min(backend_cap, Azure); duration follows

Scaling deltas (current → required)

run = concurrency actually carried backend_need = run → scale backend by max(0, run − backend_now) SIP_need = ⌈run × (1+headroom)⌉ → buy max(0, SIP_need − SIP_now) DID_need = ⌈dials_per_day / cap⌉ → procure max(0, DID_need − DID_now) Azure_need = run → quota-raise if run > Azure_now today's ceiling = min(backend_now, SIP_now, Azure_now) (binding layer)
In capped mode SIP and DIDs are sized to the backend cap, not to raw demand — you never buy channels the backend can't feed.

Fleet rollup

platform_concurrency = ⌈ (Σ tenant_concurrency) × simultaneity ⌉ platform_backend_target = platform_concurrency platform_SIP = ⌈platform_concurrency × (1+headroom)⌉ platform_DIDs = Σ tenant_DIDs (branded numbers don't pool across tenants)

Glossary

ASR / AHT
Connect rate and connected talk time. Use measured per-vertical values from ClickHouse.
Occupancy
Share of time a channel is actually on a call. 0.75–0.90 with good pacing.
CPS
Rate you may start calls. Separate from the concurrent-channel cap. Burst past it → SIP 503.
SIP headroom
Buffer of SIP channels bought above peak concurrency. Real traffic spikes above the average — pacing overshoot, retry clusters, setup/teardown overlap. Without slack, the carrier rejects the call setup (SIP 503) the instant channels are full and the call is lost. Same idea as sizing trunks for a low blocking probability, not for the mean. 20% → buy 1.2× concurrency. Applied to SIP because it is the layer that hard-rejects calls when full.
Channels vs DIDs
Channels = concurrency. DIDs = caller identities, sized by spam-safe daily volume and rotated.
Reserved + burst
Give each tenant a guaranteed floor plus a shared burst pool; backend target = Σ floors + burst pool.
A tenant's real throughput is the minimum across its backend slice, SIP channels, DID daily ceiling, and Azure budget. Provision balanced — extra SIP behind a small DID pool or a capped backend is wasted spend.
